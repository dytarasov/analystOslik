Задача по каждой колонке:
- понятное бизнес-описание (что это, как аналитик это использует); для строк-контейнеров (JSON / массив / список через разделитель) — назови формат и что лежит внутри, опираясь на примеры;
- уточни роль (id|fk|measure|dimension|timestamp|flag|free_text);
- единица измерения (unit) если применимо, иначе null;
- pii: true если персональные данные;
- value_meanings: смысл каждого значения для enum/статусов/флагов (если применимо), иначе {};
- safe_to_group_by / safe_to_filter_by: можно ли безопасно группировать/фильтровать по колонке;
- caveats: подводные камни (много null, депрекейт, неоднозначность); для практически пустых или константных колонок (null_ratio≈1 либо одно значение) — отметь это здесь как вероятный депрекейт/заглушку с низким confidence, НЕ задавай вопрос;
- suggested_aggregation: sum|avg|count|count_distinct|min|max|none;
- confidence: 0..1 — насколько ты уверен в трактовке.

ВСЕГДА заполняй описание в "columns" по каждой колонке (лучшая трактовка по фактам), даже если параллельно задаёшь вопрос.

НИКОГДА не спрашивай про то, что уже есть в фактах выше или выводится из них:
- тип данных, формат хранения/сериализации (JSON, CSV, массив, строка, «через запятую»), внутренняя структура строки, длина, кодировка;
- примеры значений, доля NULL, число уникальных, диапазон.
Всё это определи САМ по примерам, каталогу и профилю.

Спрашивай ТОЛЬКО про невыводимый бизнес-смысл, и только если глоссарий/контекст не дают ответа:
- смысл непрозрачных кодов/енумов, которых нет в каталоге значений (например, что значат коды 0/1/3);
- к какой сущности/таблице ведёт идентификатор (FK), если это не очевидно из имени;
- реальное смысловое различие двух почти одинаковых по названию колонок;
- бизнес-правило, влияющее на трактовку метрики.
Каждый вопрос обязан содержать "reason" — почему ответ нельзя вывести из фактов.
